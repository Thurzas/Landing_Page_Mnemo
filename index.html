Session 2026-03-02 — Docker opérationnel

Router pattern.
Docker en production.

Architecture de dispatch implémentée, Docker durci et opérationnel après 13 bugs de containerisation. LiteLLM stable, agent en production locale. Phase 3 : le ShellCrew peut être construit sur des fondations solides.

DESIGN · Router pattern
_route_message — 4 routes, dispatch propre
Le pipeline linéaire evaluate→retrieve→main ne permet pas d'isoler les crews d'action (shell, calendar). Un ShellCrew avec accès aux outils mémoire = risque de fuite croisée.
_route_message(eval_json, …) dispatche selon route : "conversation" (défaut silencieux), "shell", "calendar", "scheduler". web_context injecté dans tous les crews. Route inconnue → conversation sans exception
DESIGN · EvaluationCrew
Champ route + règles de déclenchement
L'évaluateur doit distinguer lecture calendrier (→ conversation) et écriture (→ calendar), commande explicite (→ shell) et "montre-moi comment faire" (→ conversation).
4 valeurs dans evaluation_tasks.yaml : règles précises, exemples, règle explicite lecture≠écriture calendrier, fallback "conversation" en cas de doute
FEATURE · Stubs Phase 3
ShellCrew · CalendarWriteCrew · SchedulerCrew
ImportError garanti si le router tente de dispatch vers un crew non déclaré. Les stubs permettent de tester le router sans implémenter les crews.
3 classes stub dans crew.py avec .run(inputs) → message "non encore implémenté". Compartimentalisation garantie : chaque crew n'aura accès qu'à ses propres outils
FEATURE · Tests
32 tests — test_router.py
Le router est un nœud critique — une régression silencieuse enverrait tous les messages vers le mauvais crew sans aucun signal d'erreur visible.
4 classes : TestParseEvalJson (JSON bruité, trailing text), TestRouteMessage (4 routes + fallback + web_context), TestPhase3Stubs, TestHandleMessage (pipeline bout en bout mocké)
DOCKER · Durcissement
cap_drop · read_only · non-root · réseau isolé
Un ShellCrew sans cloisonnement = une commande confirmée par erreur peut toucher la machine hôte. L'isolation doit être garantie avant d'implémenter les outils shell.
cap_drop: ALL + no-new-privileges + read_only: true + user: 1000:1000 + limits RAM/CPU. Réseau dual : mnemo_internal (SearXNG isolé) + mnemo_external (Ollama via host-gateway)
FEATURE · shell_whitelist.py
Whitelist de commandes pour ShellCrew
Sans whitelist explicite, le LLM pourrait proposer n'importe quelle commande. La confirmation utilisateur seule ne suffit pas si la commande est formulée de façon trompeuse.
3 familles : fichiers /data, lecture système, scripts Python dans /data. Blacklist explicite (bash, curl, sudo, pip). Validation path traversal, flags rm -rf interdits, sortie tronquée à 50 KB
FIX · LiteLLM
LiteLLM résolu — provider ollama/ natif
13 tentatives pour stabiliser LiteLLM : venv uv vide, conflit de versions, provider openai/ mal routé, venv inexistant dans l'image. L'erreur "Fallback to LiteLLM is not available" persistait malgré les installs.
Retour au pip système sans venv. litellm déclaré explicitement dans requirements.txt. crew.py restauré avec ollama/model + base_url — format natif que litellm comprend une fois proprement installé
FIX · Config
3 fixes config de fin de session
{label} dans curiosity_tasks.yaml interprété comme variable de template CrewAI → crash à chaque écriture. install.sh sans -f → compose ne trouvait pas le fichier. Traces CrewAI interactives parasitaient le terminal.
{{label}} échappé dans le YAML. COMPOSE_FILE + -f dans install.sh. CREWAI_TRACING_ENABLED=false dans .env + Dockerfile
Philosophie

Conçu pour durer.
Pas pour impressionner.

Chaque décision d'architecture a été prise pour la robustesse à long terme : mémoire éditable manuellement, zéro vendor lock-in, tests avant features, sécurité avant fonctionnalité.

📄
memory.md comme source de vérité
La mémoire reste un fichier Markdown lisible et éditable humainement. Versionnable avec git. SQLite est un index — pas la source. Les champs vides contiennent des notes d'intention pour l'agent.
🔒
100% local, 100% privé
Ollama en Docker, SQLite sur disque, aucun appel vers un service externe. Télémétrie CrewAI désactivée. Ce qui est dit à Mnemo reste sur la machine.
🧪
Python avant LLM
Réduire la surface LLM. La détection de gaps, la validation de commandes shell, les gardes-fous réseau — tout ce qui peut être fait en Python pur ne passe plus par un agent. Moins d'hallucinations, résultat garanti.
🧠
Mémoire ≠ base de données
La mémoire personnelle et les documents ingérés sont séparés physiquement mais fusionnés dans le retrieval — pour que l'agent sache toujours d'où vient chaque information.
🛡️
Sécurité avant fonctionnalité
Le ShellCrew n'existera pas avant que le conteneur soit blindé. Whitelist avant implémentation. Confirmation utilisateur sur query figée avant envoi réseau. La surface d'attaque se rétrécit à chaque phase.

Sécurité · Docker Phase 3

Cloisonné.
Avant d'agir.

Un agent capable d'exécuter des commandes système doit tourner dans un conteneur blindé. Toutes les garanties sont en place avant d'implémenter le ShellCrew.

🚫
cap_drop: ALL + no-new-privileges
Toutes les capabilities Linux supprimées. Aucune escalade de droits possible via setuid. L'agent ne peut pas monter de filesystem, ne peut pas raw-socket, ne peut rien faire hors /data.
cap_drop: [ALL]
security_opt:
- no-new-privileges:true
🔏
Filesystem read-only
Tout le filesystem est non-modifiable sauf /data (volume hôte) et deux tmpfs en mémoire. Le code source dans /app ne peut pas être modifié depuis l'intérieur du conteneur.
read_only: true
tmpfs: [/tmp, /run]
volumes: [DATA:/data]
👤
Utilisateur non-root · uid 1000
L'agent ne tourne jamais en root. Même si une vulnérabilité permettait d'exécuter du code arbitraire, les droits sont ceux d'un utilisateur lambda sans sudo.
USER mnemo # uid 1000
user: "1000:1000"
# HOME=/tmp → ChromaDB cache
⚡
Limites ressources
Une commande mal formée ou un LLM en boucle ne peut pas saturer la machine hôte. Limites RAM et CPU configurables via .env.
memory: ${MNEMO_MEMORY_LIMIT:-1g}
cpus: "${MNEMO_CPU_LIMIT:-2.0}"
📋
shell_whitelist.py
Whitelist de commandes validée avant confirmation utilisateur. 3 familles autorisées. Blacklist explicite (bash, curl, sudo). Validation path traversal. rm -rf impossible.
ALLOWED: ls cat find grep
mkdir mv cp rm touch
python (*.py dans /data)

BUG #1
Perte silencieuse de chunks
Dernier ### de chaque ## perdu lors du parsing — sous-sections entières non indexées.
Sauvegarde du chunk courant avant reset à chaque changement de section
BUG #2
Crash sur query vide
SQLite FTS5 lève OperationalError sur string vide. Pipeline entier plantait.
Guard if not query.strip(): return [] en tête de search_keyword()
BUG #3-6
Collision ID · AttributeError · Seuil MIN_LEN · AST walk doublons
Hash sur contenu seul → collision. para.style=None → crash DOCX. MIN_CHUNK_LEN=80 → fonctions courtes perdues. ast.walk() → méthodes indexées deux fois.
Hash sur position+contenu · guard triple para.style · seuil désactivé code · iter_child_nodes()
SESSIONS #7-13
7 bugs de session (CuriosityCrew + Web)
Import inexistant · _TeeStream cassait input() · update_markdown écrasait tout · labels template = faux positifs · boucle infinie style · header ### matchait alias · merge écrasait valeurs réelles
Renommage imports · suppression TeeStream · enrichissement conditionnel · _line_is_real_value() · skip lignes # · filtrage ligne par ligne
BUG #14 · Docker
install.sh ne trouve pas docker-compose.yml
Script lancé depuis /waifuclawd/ — docker compose build cherchait le fichier dans le CWD, pas dans docker/.
cd "$(dirname "${BASH_SOURCE[0]}")" + -f "${COMPOSE_FILE}" explicite dans toutes les commandes compose
BUG #15 · Docker
read_only bloquait ChromaDB (HOME=/home/mnemo)
CrewAI tente mkdir /home/mnemo/.local/share/data à l'import. read_only:true + /home non writable → PermissionError avant même le premier message.
ENV HOME=/tmp dans Dockerfile — tmpfs déjà writable, ChromaDB cache éphémère
BUG #16 · Docker
litellm manquant → ImportError
crewai seul dans requirements.txt ne tire pas litellm. CrewAI ≥0.130 l'exige pour router les appels LLM vers Ollama.
crewai[tools] + litellm[proxy] + fastapi dans requirements.txt
BUG #17 · Docker
Commande "run" absente du dispatch
CMD ["run"] dans Dockerfile, mais le if/elif dans main.py ne gérait pas la valeur "run" → "Commande inconnue : run".
Ajout de if sys.argv[1] in ("run", "start"): run() en tête du dispatch
BUG #18 · Docker
Réseau internal bloquait Ollama
internal: true sur l'unique réseau coupe tout trafic sortant, y compris vers host.docker.internal (Ollama). L'agent ne pouvait plus évaluer aucun message.
Deux réseaux : mnemo_internal (SearXNG, isolé) + mnemo_external (Ollama via host-gateway)
BUG #19 · Docker
Client ollama Python ignorait API_BASE
ollama.embeddings() se connecte toujours à localhost:11434 — ignorait API_BASE=http://host.docker.internal:11434. ConnectionError à chaque embedding.
_ollama_client = ollama.Client(host=OLLAMA_HOST) au niveau module — OLLAMA_HOST lu depuis l'env
BUG #20 · Docker
web_search non patchable dans les tests
Import local de web_search à l'intérieur de _handle_web_confirmation() — patch('Mnemo.main.web_search') trouvait AttributeError car le symbole n'existait pas au niveau module.
Import déplacé au niveau module — patch('Mnemo.tools.web_tools.web_search') en fallback pour compatibilité
BUG #21 · Docker
Venv uv vide — packages installés dans le mauvais Python
uv venv /app/venv crée le venv mais pip install (sans activation) installait dans le Python système. ENTRYPOINT pointait vers /app/venv/bin/python — vide. L'agent démarrait sans aucune dépendance.
Suppression du venv. Retour au pip système direct — simple, sans ambiguïté, packages toujours là où Python les cherche
BUG #22
{label} interprété comme variable de template
CrewAI parse les descriptions de tâches YAML et tente d'injecter toutes les {variables}. L'exemple de format - **{label}** : {réponse} dans curiosity_tasks.yaml déclenchait un crash à chaque écriture de réponse.
{{label}} et {{réponse}} — double accolades pour échapper l'interpolation CrewAI
BUG #23
install.sh sans -f → compose file introuvable
Le fix COMPOSE_FILE avait été appliqué en session mais jamais répercuté sur install.sh. docker_compose() appelait $COMPOSE_CMD "$@" sans -f → "no configuration file provided" depuis tout répertoire autre que docker/.
COMPOSE_FILE détecté au démarrage du script. docker_compose() → $COMPOSE_CMD -f "${COMPOSE_FILE}" "$@"
