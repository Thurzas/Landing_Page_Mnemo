Session v9 · 4 mars 2026

5 bugs réglés.
Scheduler complet.

Calendrier RRULE, temporal reasoning, pre-fetch calendrier, scheduling système + utilisateur — et tous les chemins read-only corrigés.

BUG #43 · ICS
Récurrences ICS ignorées — 2 événements au lieu de 20
get_upcoming_events() lisait DTSTART brut. Les événements récurrents (RRULE:FREQ=WEEKLY;BYDAY=TH) avaient un DTSTART en septembre 2025 — jamais dans la fenêtre "aujourd'hui+14j".
Expansion RRULE manuelle native — FREQ=WEEKLY/DAILY, BYDAY, UNTIL, COUNT, EXDATE. Fallback recurring_ical_events si disponible. Déduplication via (uid, date).
BUG #44-45 · Banner
Bannière et greeting affichaient 3 jours d'événements
format_startup_banner() affichait les événements passés + 14 jours futurs. Le greeting du matin listait tous les événements des 3 prochains jours.
Banner → days_until == 0 uniquement. Greeting → aujourd'hui + demain uniquement, suppression du bloc urgent_soon.
BUG #46 · Temporal
LLM ne comprenait pas "prochaine session après aujourd'hui"
format_events_for_prompt() affichait [Dans 2 jours] sans date ISO. Le LLM ne pouvait pas calculer "la prochaine occurrence après aujourd'hui".
Format enrichi : 2026-03-06 (jeudi). Instructions explicites dans conversation_tasks.yaml : "pour prochaine session X, chercher première occurrence dont date > aujourd'hui".
BUG #47 · Iterator
"Maximum iterations reached" sur toutes les questions calendrier
memory_retriever (max_iter=8) recevait needs_calendar: true → appelait GetCalendarTool → reformatait JSON → retry → explosion avant toute réponse.
Pré-fetch calendrier dans main.py avant le kickoff CrewAI. calendar_context injecté directement dans base_inputs. Instruction "n'appelle PAS get_calendar_events" dans le task YAML.
BUG #48 · Read-only FS
uncertain_cases.jsonl écrit dans /app — filesystem read-only
_UNCERTAIN_LOG = Path(__file__).parent / "uncertain_cases.jsonl" pointait vers /app/src/Mnemo/ — monté read-only dans Docker. Même problème pour router_model.joblib, training_data.jsonl.
Tous les fichiers de données redirigés vers Path("...").resolve() relatif à WORKDIR /data. router_model.joblib : cherche d'abord dans /data, fallback /app.
FEAT · Scheduler
Service scheduler complet — 4 actions, boucle 60s
Aucun service de planification n'existait. Les tâches différées ("rappelle-moi demain") n'étaient pas possibles.
scheduler.py service Docker séparé. tools/scheduler_tasks.py : CRUD SQLite + compute_next_run() + mirror tasks.md + bootstrap_system_tasks(). Actions : briefing / weekly / deadline_alert / reminder.
FEAT · SchedulerCrew
SchedulerCrew — du langage naturel à la DB
Le stub SchedulerCrew retournait "non implémenté". Impossible de créer des rappels depuis la conversation.
Agent dédié + prompt structuré : détecte one_shot/recurring, calcule trigger_at ou cron_expr, parse le JSON de retour, appelle create_task() ou cancel_task(), retourne une confirmation en français naturel.
ARCH · Routing
Détection scheduler — 20 keywords + arbitrage post-LLM
Les messages type "rappelle-moi demain" étaient routés vers conversation — le LLM répondait mais ne créait aucune tâche.
_detect_scheduler_intent() : 20 keywords (rappelle-moi, planifie, chaque lundi…). Arbitrage post-LLM : si LLM dit conversation et keywords détectent scheduler → correction. Seuil ML 0.75 vs 0.92 pour shell.
FEAT · DX
mnemo.sh — raccourci de lancement
Lancer Mnemo nécessitait de mémoriser les commandes docker compose run --rm. Friction à chaque session.
mnemo.sh à la racine. Commandes : run / scheduler / stop / briefing / weekly / deadline / logs / ingest / status / help. Tip WSL2 : alias mnemo généré dans install.sh.

Docker · v9

2 services.
1 volume partagé.

Le scheduler tourne en background dans son propre container. Même sécurité que le service principal. /data est le seul point de contact entre les deux.

🤖
mnemo — service principal
Session interactive CLI. read-only, cap_drop: ALL, uid 1000. Redémarre jamais automatiquement (restart: "no") — session intentionnelle.
docker compose run --rm mnemo
./mnemo.sh
⏰
mnemo-scheduler — service background
Tourne en daemon. Même sécurité (read-only, cap_drop: ALL, uid 1000). Volume /data partagé. DATA_PATH=/data forcé pour éviter résolution depuis WORKDIR.
./mnemo.sh scheduler
restart: on-failure:3
📁
/data — source de vérité partagée
Volume monté dans les deux containers. Contient : memory.db, memory.md, sessions/, briefing.md, weekly.md, tasks.md, uncertain_cases.jsonl, router_model.joblib.
volumes: ${DATA_DIR:-./data}:/data
